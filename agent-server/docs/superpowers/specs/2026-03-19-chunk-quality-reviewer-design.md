# Chunk Quality Reviewer Skill — 设计文档

**日期：** 2026-03-19
**状态：** 已批准

---

## 概述

`chunk-quality-reviewer` 是一个 Claude Code skill，通过对话式诊断帮助用户评估知识库 chunk 的质量——是否最大化了检索有效信息，同时最小化了噪音。

用户提供一个包含 chunks 的 JSON 文件和需要读取的字段路径，skill 先了解使用场景，再对 chunks 进行多维度分析，最终输出问题清单、整体评估和改进建议。

---

## 输入

- **JSON 文件路径**：包含 chunks 的文件
- **字段路径**：用户指定需要读取的字段（如 `node_output.final_chunks[*].content` 和 `.meta`）

---

## 工作流程

### Phase 1：情境问诊

读取 chunks 后，先向用户提出两个问题（一次一个）：

**Q1 — 查询类型**
> 这个知识库主要用来回答哪类问题？
> - 事实查询（如某指标的限量值是多少）
> - 流程指导（如某检验方法的操作步骤）
> - 对比分析（如不同产品类别的指标对比）
> - 综合问答（多条款联合推理）

**Q2 — 文档结构类型**
> 文档主要是什么结构？
> - 条款型（法规/标准，有编号条款）
> - 步骤型（操作手册、检验规程）
> - 表格密集型（大量指标表格）
> - 混合型

用户的回答决定各评估维度的**侧重权重**：

| 场景组合 | 重点维度 |
|---------|---------|
| 事实查询 + 条款型 | 自包含性、数值完整性、条款边界 |
| 流程指导 + 步骤型 | 语义完整性、结构完整性（步骤连续） |
| 对比分析 + 表格密集型 | 结构完整性（表格）、主题聚焦性 |
| 综合问答 + 混合型 | 全维度均衡评估 |

---

### Phase 2：多维度 Chunk 分析

对所有 chunks 按以下六个维度逐一检查：

#### 2.1 自包含性（Self-Containedness）
- **定义**：chunk 脱离文档后，能否独立回答某个具体问题
- **问题信号**：
  - 出现"如前所述"、"上述要求"、"见第X条"等指向外部内容的表达
  - 以代词或连词开头（"该产品"、"此外"、"因此"）但所指内容不在本 chunk 内
- **严重程度**：高（直接影响检索结果的可用性）

#### 2.2 语义完整性（Semantic Completeness）
- **定义**：chunk 是否在完整的语义边界处截断，没有切断句子、段落或逻辑单元
- **问题信号**：
  - chunk 以未完结的条件句结尾（"如果…"但无"则…"）
  - 编号列表只有部分项目（"1. … 2. …"但后续项在其他 chunk）
  - 步骤序列中间截断
- **严重程度**：高

#### 2.3 信息密度（Information Density）
- **定义**：chunk 中有效信息（事实、规则、数据）与噪音（模板文字、格式残留、页眉页脚）的比值
- **问题信号**：
  - chunk 内容几乎全是样板文字（"本章节将介绍……"）
  - 仅包含章节标题，无实质内容
  - 包含页码、版权声明、目录条目
- **严重程度**：中（影响检索精度）

#### 2.4 主题聚焦性（Topical Coherence）
- **定义**：一个 chunk 只讨论一个明确的主题，不混杂多个不相关的概念
- **问题信号**：
  - 单个 chunk 同时涵盖理化指标和微生物指标（不同检测类别）
  - 将定义条款与限量指标合并在同一 chunk
- **严重程度**：中（影响检索召回的准确性）

#### 2.5 上下文可理解性（Contextual Interpretability）
- **定义**：chunk 内的专有名词、缩写、代词有足够的上下文支撑理解
- **问题信号**：
  - "该物质"、"本标准"的所指对象不在 chunk 内
  - 使用未定义的缩写或领域术语
- **严重程度**：中

#### 2.6 结构完整性（Structure Integrity）
- **定义**：表格、公式、步骤等结构化内容保持完整，不被截断
- **问题信号**：
  - 表格只有数据行，缺少表头（列名/单位）
  - 数学公式与其变量定义分散在不同 chunk
  - 图注与对应图表编号分离
- **严重程度**：高（结构破坏后内容完全无法理解）

---

### Phase 3：三层输出

#### 3.1 问题清单

按严重程度分级列出，每条问题包含：chunk 标识、问题维度、具体表现。

```
## 问题清单

### 严重问题（需立即修复）
- Chunk #12 [结构完整性]：表格被截断，缺少表头，行列关系不可读
- Chunk #5  [语义完整性]：步骤序列在第3步中断，第4-6步在 Chunk #6

### 中等问题（建议修复）
- Chunk #7  [自包含性]：开头"此外，上述要求..."，上文不在本 chunk 内
- Chunk #19 [主题聚焦性]：同时包含理化指标和包装要求，主题混杂

### 轻微问题（可选优化）
- Chunk #3  [信息密度]：仅包含章节标题"4.2 理化指标"，无实质内容
- Chunk #21 [信息密度]：包含页脚文字"GB 2762-2022 第5页"
```

#### 3.2 整体质量评估

各维度评分（1-5分）及简要说明，聚焦最主要的问题。

```
## 整体质量评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 自包含性 | 3/5 | 约30%的chunk存在悬空引用 |
| 语义完整性 | 4/5 | 边界切分基本合理，少量步骤被截断 |
| 信息密度 | 4/5 | 少量标题类噪音chunk |
| 主题聚焦性 | 3/5 | 部分chunk混合了不相关主题 |
| 上下文可理解性 | 3/5 | 代词悬空问题较常见 |
| 结构完整性 | 2/5 | 表格处理是最大问题 |

**总体评分：3.2/5**
**主要瓶颈：表格结构完整性 + 自包含性**
```

#### 3.3 改进建议

针对发现的具体问题，给出可操作的切分策略调整建议，按优先级排序。

```
## 改进建议

### 最优先
1. **表格处理策略**：表格整体保留在一个 chunk；若表格过大，
   按逻辑行分组（如按食品类别），每组重复表头行

### 次优先
2. **上下文前缀注入（Contextual Retrieval）**：对有悬空引用的
   chunk，在嵌入前自动添加50-100词的上下文说明（所在章节、
   文档位置），降低检索失败率

3. **步骤序列完整性**：检验方法类内容以"完整方法"为切分单元，
   不按固定大小截断

### 可选优化
4. **合并空洞 chunk**：将仅含标题的 chunk 与其正文合并
5. **清洗噪音内容**：在切分前过滤页眉页脚、目录条目
```

---

## 关键设计原则

1. **场景优先**：评估标准根据查询类型和文档结构动态调整权重，避免泛泛而谈
2. **严重程度分级**：区分"必须修复"和"可选优化"，帮助用户聚焦资源
3. **具体可操作**：改进建议直接对应切分策略的调整方式，而非抽象原则
4. **覆盖结构化文档特殊性**：表格、公式、条款编号等结构化内容有专项检查

---

## 不在范围内

- 自动运行 RAGAS/LlamaIndex 等评估框架（需要 Python 环境和查询测试集）
- 生成量化的 Context Precision/Recall 指标（需要黄金标准测试集）
- 自动修复 chunks（仅评估和建议，不修改数据）
