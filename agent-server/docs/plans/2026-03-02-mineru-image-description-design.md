# MinerU 解析图片的 VLM 描述追加 — 设计

> 在 PDF 经 MinerU 得到 Markdown 后、写入知识库前，用多模态 LLM 对每张图生成一段描述并追加在图片引用后，使图片语义可被检索。

## 1. 目标与范围

**目标**：在 PDF 经 MinerU 得到 Markdown 后、写入知识库（切分 + 向量化）之前，增加一步「图片理解」：用多模态 LLM 对 Markdown 中引用的每张图生成一段文字描述，并**追加**在该图片引用后（保留原 `![](...)`），使后续切分与检索都能利用图片语义。

**范围**：
- **新增**：在取得 `md_content` 之后、调用 `split_heading_from_markdown` 之前，执行「Markdown 内图片 → VLM 描述 → 在原位追加描述」；可选配置项（是否启用、模型、超时等）。
- **不改**：切分策略、向量库/Neo4j 写入的接口与格式；知识库仍只存文本，不做图像存储或多模态向量。
- **失败降级**：单图 VLM 失败时不追加描述（或追加占位文案），打日志，不阻断整次导入。

**依赖**：接入支持「图 + 文本输入、文本输出」的多模态 LLM；可复用现有 vision 封装或在本模块内做最小封装。

---

## 2. 流程位置、输入输出与组件

**流程位置**：在 MinerU 适配器内，在读取到 `.md` 之后、删除临时目录之前：若启用图片描述，则调用「图片描述」模块（传入 `md_content` 与 `output_dir` 作为图片根路径），得到追加描述后的 Markdown，再返回并删除临时目录。这样 `pdf_to_markdown` 仍返回 `str`，调用方（`import_pdf_via_mineru`）无需改动。

**输入**：当前 Markdown 字符串 `md_content`；图片根路径 `images_base_path`（即 MinerU 的 output_dir，用于解析 `![](...)` 中的相对 path）。

**输出**：新的 Markdown 字符串，在每个 `![](...)` 后追加对应描述（如 `\n\n图注：...`），原引用保留；无图片或未启用时返回原 `md_content`。

**组件**：
1. **图片发现**：从 Markdown 解析所有 `![](path)` / `![alt](path)`，得到 path 列表；在 `images_base_path` 下解析为本地绝对路径并读入图片 bytes（无法解析则跳过并打日志）。
2. **VLM 调用**：对每张图（+ 可选简单 prompt）调用多模态 LLM，得到一段话；支持单图 + 固定/可配置 prompt。
3. **Markdown 替换**：按出现顺序，将每个 `![](...)` 替换为「原引用 + 换行 + 追加描述」；单图失败则该处仅保留原引用或占位。

---

## 3. 配置与 VLM 接入

**配置项**（如 `app/core/config.py`）：
- `MINERU_IMAGE_DESCRIPTION_ENABLED: bool = True`，为 `False` 时不执行图片描述。
- `MINERU_IMAGE_DESCRIPTION_MODEL` 或复用现有 vision 模型配置；若单独配置则需明确模型/端点。
- 单图超时（如 30s）、失败重试次数（如 1）；超时/失败则该图不追加或占位。
- 可选：单文档最多处理图片数上限（如 50）。

**VLM 接入**：复用项目已有「多模态/视觉」调用；若无则本模块内最小封装：按配置调用 API（OpenAI vision、DashScope、Ollama 视觉等），输入为单图 + prompt，输出为纯文本作为图注。

---

## 4. 错误处理与测试

**错误处理**：
- 无图片引用：直接返回原 `md_content`。
- 某 path 在 base_path 下不存在/不可读：跳过该图，打日志，不阻断。
- VLM 失败/超时：该图不追加或占位，打日志，继续其余图。
- 功能关闭或未配置 VLM：跳过本步，返回原 `md_content`。

**测试**：
- 单元测试：输入含 1～2 个 `![](...)` 的 Markdown + mock base_path，mock VLM 返回固定描述，断言输出中每个引用后追加了描述且原引用保留；无图片时输出等于输入；可选 mock VLM 异常断言降级。
- 集成/手动：含图 PDF 跑一次导入，检查 chunk 中含图注；不强制 CI。
